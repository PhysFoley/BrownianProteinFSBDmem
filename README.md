# Membrane-Coupled Protein Lattice Simulations (HOOMD-blue)

This repository contains code for simulating flexible protein lattices (e.g. clathrin
triskelia, or arbitrary rigid-body assemblies output by [NERDSS](https://github.com/JohnsonBiophysicsLab/NERDSS))
coupled to a deformable membrane. The membrane is treated in the small-gradient
(Monge-gauge) Helfrich approximation and propagated in Fourier space using
overdamped (Oseen-hydrodynamic) Brownian dynamics, following Lin & Brown, *Phys.
Rev. Lett.* (2004). Protein degrees of freedom are simulated as rigid bodies in
[HOOMD-blue](https://hoomd-blue.readthedocs.io/) and coupled to the membrane through
harmonic tether bonds between designated particles and the local membrane height.

See the comments in `default_params` at the top of each driver script for a description of every configurable parameter.

## Contents

| File | Purpose |
|---|---|
| `run_arbitrary_geom.py` | Driver script: simulates a lattice of triskelia given explicit initial center-of-mass coordinates and orientations, coupled to a membrane. |
| `run_nerdss_config.py` | Driver script: simulates a protein complex produced by a NERDSS simulation (after preprocessing), coupled to a membrane. |
| `prepare_nerdss_complex.py` | Preprocessing script that takes a raw NERDSS complex snapshot and `.mol` files and prepares the JSON input consumed by `run_nerdss_config.py`, including orienting the complex relative to the membrane and interactively collecting bonded-potential parameters. |
| `membrane.py` | `Membrane` class implementing the Fourier-space Helfrich/Brownian membrane propagator, the HOOMD custom force that couples tethered particles to the membrane, and `MembraneWriter` for trajectory output. |
| `sys_setup.py` | Helper routines for building initial HOOMD simulation states, parsing NERDSS `.mol` files, and converting a NERDSS reaction-diffusion snapshot into a HOOMD rigid-body system with the correct bond/angle/dihedral potentials. |
| `examples/` | One example parameter/input file for each driver script (see below). |

## Unit system

All scripts use a consistent unit system:

- length — nanometers (nm)
- time — nanoseconds (ns)
- energy — kT at 310 K

Derived units follow from these choices (e.g. mass ≈ 2.578 MDa).

## Requirements

- Python 3.10+ (uses PEP 701 f-string/dict syntax in the driver scripts)
- [HOOMD-blue](https://hoomd-blue.readthedocs.io/) (with `hoomd.md`, rigid-body constraints, and CPU device support)
- [gsd](https://gsd.readthedocs.io/)
- NumPy, SciPy

A CPU device is used by default (`hoomd.device.CPU()`); GPU support would require
adapting the custom membrane force in `membrane.py`, which currently runs on the CPU
local snapshot/force arrays.

## The membrane model

`Membrane` (in `membrane.py`) represents the membrane height field `h(x,y)` on an
`N x N` periodic lattice spanning the simulation box (`N` must be odd). Internally
the height field is stored and updated in Fourier space (`hk`), where the Helfrich
bending (`kappa`) and tension (`sigma`) contributions to the equation of motion are
diagonal. Each timestep:

1. External forces (from particles tethered to the membrane) are Fourier-transformed
   and added to the deterministic Helfrich force.
2. Thermal noise is drawn per-mode with amplitude set by the Oseen mobility
   `lambda(k) = 1/(4*eta*|k|)` (`eta` = cytosol viscosity), following the
   hydrodynamically-correct membrane fluctuation theory of Lin & Brown.
3. Modes are integrated with an explicit Euler-Maruyama step, redundant degrees of
   freedom are projected out, and conjugate-symmetry is enforced so that `h(x,y)`
   remains real.

Particles registered as "membrane-bonded" are tethered to the local membrane patch
beneath them via a harmonic bond (stiffness `kbond_pm`/`kbond_cm`, rest length
`pm_rest_len`/`bind_l`); the reaction force is deposited back onto the membrane as an
external force density. This coupling is implemented as a `hoomd.md.force.Custom`
subclass (`MembraneForce`) appended to the integrator's force list, so the membrane
update is advanced once per simulation step alongside the particle integrator.
`MembraneWriter` periodically appends the real-space height field to a JSON
trajectory file (`mem_traj.json`) alongside metadata (box size, lattice number,
lattice spacing, timestep).

## Workflow 1: arbitrary initial geometry (`run_arbitrary_geom.py`)

Use this driver to simulate a lattice of clathrin triskelia (or a rigid variant of
the same monomer geometry) starting from a user-specified set of initial positions
and orientations, e.g. flat, buckled, or spherical-cap lattices generated by the
helper constructors in `sys_setup.py`.

**Inputs**

- A JSON parameter file (see `examples/arbitrary_geom` for a template) specifying membrane and
  particle-mechanical parameters (`kr`, `ktheta`, `komega`, `kappa`, `eta`, box size,
  timestep, run length, etc.)
- `com_coords_file`: JSON file of per-triskelion center-of-mass coordinates
- `rot_vectors_file`: JSON file of per-triskelion rotation vectors

**Run**

```bash
cd examples/arbitrary_geom/
python ../../run_arbitrary_geom.py params.json
```

Any parameter omitted from the JSON file falls back to the defaults defined at the
top of the script (`default_params`).

**Output** (written to `out_<jobname>/`)

- `trajectory.gsd` — particle trajectory (positions, orientations, topology)
- `mem_traj.json` — membrane height-field trajectory
- console progress messages at the interval set by `progress_out_incr`

## Workflow 2: NERDSS-derived complexes (`run_nerdss_config.py`)

Use this driver to continue a structure that originated from a NERDSS
reaction-diffusion simulation (e.g. a stochastically assembled lattice) inside the
membrane-coupled HOOMD framework.

### Step 1 — preprocess with `prepare_nerdss_complex.py`

NERDSS complex data is provided by setting the `bondedComplexWrite` parameter
in the NERDSS simulation parameters file. See the NERDSS user guide for more
information.

NERDSS output must first be converted into the bonded-complex JSON format consumed
by `run_nerdss_config.py`. This is done with:

```bash
python prepare_nerdss_complex.py <nerdss_frame.json> <complex_index> <membrane_site_1> [<membrane_site_2> ...]
```

- `<nerdss_frame.json>`: a NERDSS output file containing a list of complexes (each
  with `names`, `coords`, `rotations`, `bond_types`, and `bonds`)
- `<complex_index>`: index of the complex within that file to prepare
- `<membrane_site_*>`: name(s) of the interface site(s) on each molecule that anchor
  it to the membrane

For the provided example, this looks like

```bash
cd examples/nerdss_config/
python ../../prepare_nerdss_complex.py nerdss_complexes.json 0 m2muh
```

The corresponding `.mol` file(s) for every molecule type in the complex must be
present in the working directory (NERDSS molecule-template format; the center of
mass must be the first listed site).

This script:

1. Locates every particle bearing one of the specified membrane-binding site names
   and rigidly reorients the whole complex so that the binding sites lie in a plane
   parallel to (and just above) the membrane.
2. Opens an interactive Matplotlib 3D plot of the reoriented complex for visual
   inspection.
3. Prompts interactively (stdin) for the membrane-tether parameters
   (`bind_l`, `k_mem`) and, per bond type present in the complex, the harmonic/angular/
   dihedral force constants (`k_sigma`, and optionally `k_theta1`, `k_theta2`,
   `k_phi1`, `k_phi2`, `k_omega` — only for angles that are defined in the input file).
4. Ensures molecule/site ordering within each bond record is consistent with the
   corresponding `bond_types` entry.
5. Writes `processed_<nerdss_frame>.json`, ready to be referenced from a
   `run_nerdss_config.py` parameter file via `complex_file`.

### Step 2 — run the simulation

```bash
cd examples/nerdss_config/
python ../../run_nerdss_config.py params.json
```

The parameter file must set `complex_file` to the JSON produced above. Membrane
tether stiffness (`kbond_pm`) and rest length (`bind_l`) are read directly from that
file and do not need to be duplicated in the parameter file. Setting
`"rigid_assembly": true` treats the whole NERDSS complex as a single rigid body
(useful for pre-assembled, non-deformable lattices) instead of a network of
individually flexible rigid monomers.

**Output** is written to `out_<jobname>/` in the same form as Workflow 1
(`trajectory.gsd`, `mem_traj.json`, and an `init.gsd` snapshot of the assembled
initial state).

## Example inputs

The `examples/` directory contains one ready-to-run parameter file for each driver
script (and, for `run_arbitrary_geom.py`, matching `com_coords_file` /
`rot_vectors_file` inputs; for `run_nerdss_config.py`, a preprocessed complex JSON
and its associated `.mol` files).

## Notes on reproducibility

- Both driver scripts create a new output directory `out_<jobname>`; if that
  directory already exists, a timestamp is appended to `jobname` automatically so
  existing results are never overwritten.
- Each run draws a fresh HOOMD RNG seed at random (`scipy.stats.uniform`), so
  repeated runs of the same parameter file are not bitwise-reproducible unless the
  seed is fixed manually.
- The membrane lattice size `mem_N` must be odd (enforced by `Membrane.__init__`).
